# Game Log Buried Point Report

## Overview

接收三类材料并产出标准测试文档：
- 埋点需求文档（通常 `.xlsx`）
- 游戏运行日志（文本日志、JSON 行日志、检索结果）
- 输出格式要求（通常 `.docx` 样式）

先提取“需求侧期望埋点”，再提取“日志侧实际上报”，最后按固定模板输出结论、证据和建议。

## Input Contract

执行前确认以下输入：
- 需求文档：优先读取 `.xlsx`，至少包含事件名、事件参数、触发时机或业务说明。
- 运行日志：至少包含事件名（如 `name`）和关键字段（如 `user_id`、时间戳、`data`）。
- 必填参数：目标 `user_id`（字符串精确匹配）。
- 输出目标：若用户指定 `.docx`，按文档章节结构组织内容；未指定时默认按 `references/output-template.md`。

输入不完整时，必须在报告中增加“缺失信息说明”，并将无法定性的条目标记为“待确认”。

## Workflow

按顺序执行：
1. 解析 `.xlsx` 需求文档，建立期望埋点清单（事件级 + 字段级）。
2. 对运行日志做筛选：先按目标 `user_id` 精确匹配，再按事件名、时间窗、业务 ID 缩小范围。
3. 做事件级比对：缺失、冗余、疑似同义映射。
4. 做字段级比对：字段名、类型、必填、取值、口径一致性。
5. 汇总问题严重级别（S1/S2/S3）并输出修复建议。
6. 按固定章节生成最终测试输出文档内容（目标为 `.docx` 结构）。

## Log Filtering Rules

执行日志筛选时遵循以下规则：
- 第一步必须按输入 `user_id` 做字符串精确匹配；不匹配记录直接排除，不纳入分析。
- 若日志缺失 `user_id` 字段，直接排除，并在问题清单记录为“日志字段缺失(user_id)”。
- 在通过 `user_id` 过滤后，再按事件名过滤（例如 `cityplot_regionunlock`、`cityplot_event`）。
- 再按时间窗口、业务对象 ID（如 `region_id`、`plot_id`）缩小范围。
- 对“应无打点”的场景，显式记录“未检出事件”作为负向证据。
- 每条结论都绑定至少一条日志证据（行号、时间戳、原文片段三者至少两项）。

## XLSX Parsing Rules

需求文档为 `.xlsx` 时：
- 优先读取“打点需求”工作表；其他报表类工作表用于补充口径，不直接替代埋点定义。
- 将“记录事件/事件参数”相关内容转为标准结构：
  - `event_name`
  - `trigger_condition`
  - `field_name`
  - `field_rule`
  - `expected_value_or_range`
- 当单元格存在合并或跨行描述时，继承上文语义并在结果中标注“推断映射”。
- 若需求侧包含账号范围或账号条件，需与输入 `user_id` 同步校验并在报告中体现是否命中。

## DOCX Output Rules

输出目标为 `.docx` 时遵循：
- 章节顺序固定，来源于 `references/output-template.md`。
- 保留“测试准备/打点需求汇总/事件打点行为分析/问题清单/结论与建议”主干结构。
- 每个事件章节至少包含：测试场景、日志查询条件、结果证据、判定结论。
- 图证缺失时不虚构截图，使用“截图待补”占位并保持结论可追溯。

## Quality Gate

交付前逐项检查：
- 每个问题是否有证据。
- 每条修复建议是否映射到具体事件或字段。
- 严重级别定义是否一致。
- 汇总结论是否与明细冲突。
- 输出是否符合 `.docx` 章节化表达（即使当前以文本先展示，也要可直接落为 `.docx`）。
- 账号匹配性统计是否完整（匹配数量、排除数量、缺失 `user_id` 数量）。
