## 示例 1：资源消耗型活动

```markdown
**需求确认项：**
- 已确认：单次捐献消耗100粮草；积分达标可领奖。
- 待确认：每日捐献上限、奖励是否通过邮件补发。
- 继承逻辑：无。
- 基于经验补充的风险：重复点击导致多扣资源或重复领奖。

**测试用例导入表：**

| 标题* | 所属分组 | 维护人 | 前置条件 | 步骤描述 | 预期结果 | 用例等级 | 用例类型 |

| 单次捐献粮草后积分正常增加 | 粮草筹备|捐献玩法 |  | 玩家拥有至少100粮草，活动处于进行中 | 【1】进入活动页面\n【2】点击“捐献”按钮\n【3】观察粮草与积分变化 | 【1】成功进入活动页面\n【2】捐献操作成功\n【3】粮草扣减100，积分按规则增加，页面提示成功 | P1 |  |
| 粮草不足时点击捐献 | 粮草筹备|捐献玩法 |  | 玩家粮草少于100，活动处于进行中 | 【1】进入活动页面\n【2】点击“捐献”按钮 | 【1】成功进入活动页面\n【2】操作失败，不增加积分，粮草不变，并提示资源不足 | P1 |  |

**需要补充的信息：**
- 每日捐献上限
- 奖励是否支持邮件补发
```

## 示例 2：联盟协作型活动

```markdown
**需求确认项：**
- 已确认：成员可发起或加入集结；存在个人贡献和联盟总进度奖励。
- 待确认：每日发起次数上限、退出联盟后的奖励归属。
- 继承逻辑：联盟职位权限沿用通用联盟系统规则。
- 基于经验补充的风险：多人并发加入同一集结导致人数、奖励或状态异常。

**测试用例导入表：**

| 标题* | 所属分组 | 维护人 | 前置条件 | 步骤描述 | 预期结果 | 用例等级 | 用例类型 |
|------|------|------|------|------|------|------|------|
| 联盟成员发起集结任务 | 集结远征|发起集结 |  | 玩家已加入联盟，活动进行中 | 【1】进入联盟活动页\n【2】选择目标\n【3】点击“发起集结” | 【1】成功进入活动页\n【2】目标可正常选择\n【3】集结创建成功，显示剩余集结时间和参与人数 | P1 |  |
| 两名成员同时加入同一集结 | 集结远征|加入集结 |  | 已存在一个可加入的集结，两个成员同时在线 | 【1】两名成员同时点击“加入集结”\n【2】观察参与人数、状态和结算结果 | 【1】参与人数与实际上阵人数一致\n【2】无重复占位\n【3】后续结算正常 | P0 |  |
| 集结战斗结算前断线后重新连接 | 集结远征|战斗结算 |  | 玩家已加入集结，战斗即将结算 | 【1】在结算前断开网络\n【2】重新连接并返回活动页\n【3】查看贡献和奖励状态 | 【1】重连后显示正确贡献值\n【2】奖励状态与服务端一致\n【3】无重复结算 | P1 |  |
| 跨天后任务次数按规则刷新 | 集结远征|次数刷新 |  | 玩家当天剩余次数为1，接近00:00 | 【1】刷新前查看剩余次数\n【2】等待跨天\n【3】刷新活动页面 | 【1】次数按规则重置\n【2】已完成进度不丢失\n【3】已领奖状态不丢失 | P1 |  |

**需要补充的信息：**
- 每日发起次数上限
- 退出联盟后的奖励归属
```

## 使用要求

- 示例用于约束输出粒度，不要机械复制示例模块或步骤。
- 当需求不完整时，保留模板中的“待确认”与“基于经验补充的风险”。
- 当关键规则来自旧玩法复用、通用系统或历史活动时，补充“继承逻辑”。
- 当用户只要求局部更新时，只输出受影响模块的确认项和导入表行。
- 默认不输出“测试类型”和“需求依据”等内部分析列，除非用户明确要求补充。
- 当用户明确要求总结中途问题或技能改进点时，再输出“执行中遇到的问题”“对 skill/流程的建议”。
- 如果用户要求导入测试管理平台，再按对方字段格式扩展列名，不要擅自发明平台字段。
