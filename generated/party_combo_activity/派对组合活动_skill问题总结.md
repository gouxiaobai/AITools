# 派对组合活动测试用例生成问题总结

## 遇到的问题
- SKILL.md 默认读取时出现中文乱码：首次按普通文本读取 skill 文件时，正文中文出现乱码，需要显式指定 UTF-8 或通过支持编码探测的读取方式。 这会直接影响技能可用性，建议把 skill 文件统一为 UTF-8 with BOM，或在说明里明确 Windows PowerShell 的读取方式。
- Windows 非 ASCII 路径下的 Python 临时脚本读取 Excel 失败：尝试用 `python -` 直接读取 `E:` 盘中文路径 Excel 时，路径被 PowerShell/STDIN 链路替换成问号导致 `OSError: [Errno 22] Invalid argument`。 这说明 skill 在 Windows 中文目录下需要专门的路径处理建议，例如优先使用脚本文件、`pathlib` 搜索或 PowerShell COM/对象传递。
- 输入材料混合度高，但 skill 缺少优先级策略：当前目录同时包含 Markdown、Excel、图片、录屏、子目录截图。skill 没有给出“先读哪类材料、何时停止补读、何时把图片只当辅助证据”的执行优先级。 本次只能人工决定以 `派对活动.md` 为主事实来源。
- 组合活动场景缺少专门的拆分模板：现有参考更偏单玩法活动。遇到“组合入口 + 多子活动 + 通用逻辑继承”的场景时，需要额外人工决定哪些测试点属于组合层、哪些属于子活动层、哪些属于通用系统回归。
- 事实、假设、继承逻辑没有结构化置信度输出：像 BP 这种“挪英雄BP活动”的需求，很多规则是继承通用逻辑而非在当前文档明写。skill 目前只能在文字里标注待确认，没有统一的事实/假设/继承来源结构，后续复核成本较高。
- 缺少自动沉淀“本次执行问题”的固定输出块：用户明确要求总结处理中遇到的问题，这类复盘对迭代 skill 很有价值，但当前 skill 模板没有内置“执行问题总结/skill 建议”章节。

## 对 skill 的建议
- 在 skill 中新增“Windows 中文路径”附录，明确推荐的读取方式、PowerShell 编码参数和 Python 脚本文件方案。
- 新增“多源材料优先级”规则：优先 Markdown/文本化需求，其次结构化 Excel，再用图片/录屏做表现校验，不要求一次性穷尽全部附件。
- 新增“组合活动拆解模板”，输出层级固定为：组合入口层、子活动层、复用通用系统层、结束/补发层。
- 在模板中加入“事实来源/待确认/继承逻辑/经验风险”四段式结构，必要时对每条待确认附来源文件名。
- 把“执行中遇到的问题”和“对 skill 的建议”做成可选标准章节，方便每次任务反哺 skill。
