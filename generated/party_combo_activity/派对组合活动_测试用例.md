# 派对组合活动测试用例

## 需求确认项

### 已确认
- 组合活动主界面新增一级入口 id7，入口展示倒计时、数字红点和 NEW 标签。
- 组合活动页面固定包含打靶、拼图、BP 3 个子活动，固定排序，默认打开打靶页签。
- 打靶活动包含预告页、主页状态切换、大奖自选、手动射击、自动射击、奖励记录、进度奖励、排行榜、礼包和道具来源弹窗。
- 打靶奖池由 1 个自选大奖和 11 个普通奖励组成，普通奖励不放回，存在大奖保底。
- 打靶活动结束后需补发未手动领取的打靶奖励、未领取的进度奖励和排行榜奖励。
- 拼图活动包含单块拼图奖励、行列宝箱奖励和全完成大宝箱奖励，奖励只能领取 1 次。
- 拼图未领取奖励需在活动结束后通过邮件补发。
- BP 活动沿用通用 BP 逻辑，并新增任务奖励道具，任务领取提示改为通用道具获得界面。
- BP 未领取奖励需在活动结束后通过邮件补发。

### 待确认
- 组合活动总入口与 3 个子活动的最终开启关系未明确：是统一开启，还是按各自城堡等级分别显示锁态/隐藏态。
- 主界面 NEW 标签的消失规则未明确，例如首次进入后消失、首日消失，还是活动期常驻。
- 打靶“进度值+1”在文档中同时出现“完成1轮+1”和“每次抽中大奖+1”，需要确认是否完全等价。
- 打靶奖励记录弹窗的三种状态命名、排序规则、是否支持一键领取未明确。
- 自动射击“需要的道具数量（考虑权重和保底）”的精确定义未在文档中展开，建议确认服务端计算口径。
- 排行榜刷新频率、同分同毫秒达成时的兜底排序规则未明确。
- 打靶活动道具过期逻辑在文档中标注“BM 是否已有”，需要确认通用实现和邮件/回收口径。
- 拼图完整布局、全部 boxId 和 unlockPuzzleId 映射在摘要文档中未展开，需要补充完整配表验证。
- BP 继承通用逻辑的边界未明确，尤其是付费解锁、等级直购、无限宝箱、任务领取次数限制的继承规则。

### 基于经验补充的风险
- 组合入口红点、页签红点、子活动内部红点三层聚合，容易出现已领取后残留或提前消失。
- 打靶自动射击存在多次扣道具、多次发奖、断线重试幂等和保底边界计算错误的高风险。
- 打靶奖励记录手动领取与活动结束邮件补发并存，存在重复发放风险。
- 打靶排行榜按轮次和达成时间双重排序，容易出现刷新延迟或排序不稳定。
- 拼图最后1块领奖与行列宝箱解锁存在动画和状态串联，高风险点在服务端已解锁但前端未刷新。
- BP 在复用通用逻辑的基础上增加任务奖励道具，容易出现积分发了但道具未发，或次数限制只作用于一类奖励。

## 执行中遇到的问题
- SKILL.md 默认读取时出现中文乱码：首次按普通文本读取 skill 文件时，正文中文出现乱码，需要显式指定 UTF-8 或通过支持编码探测的读取方式。 这会直接影响技能可用性，建议把 skill 文件统一为 UTF-8 with BOM，或在说明里明确 Windows PowerShell 的读取方式。
- Windows 非 ASCII 路径下的 Python 临时脚本读取 Excel 失败：尝试用 `python -` 直接读取 `E:` 盘中文路径 Excel 时，路径被 PowerShell/STDIN 链路替换成问号导致 `OSError: [Errno 22] Invalid argument`。 这说明 skill 在 Windows 中文目录下需要专门的路径处理建议，例如优先使用脚本文件、`pathlib` 搜索或 PowerShell COM/对象传递。
- 输入材料混合度高，但 skill 缺少优先级策略：当前目录同时包含 Markdown、Excel、图片、录屏、子目录截图。skill 没有给出“先读哪类材料、何时停止补读、何时把图片只当辅助证据”的执行优先级。 本次只能人工决定以 `派对活动.md` 为主事实来源。
- 组合活动场景缺少专门的拆分模板：现有参考更偏单玩法活动。遇到“组合入口 + 多子活动 + 通用逻辑继承”的场景时，需要额外人工决定哪些测试点属于组合层、哪些属于子活动层、哪些属于通用系统回归。
- 事实、假设、继承逻辑没有结构化置信度输出：像 BP 这种“挪英雄BP活动”的需求，很多规则是继承通用逻辑而非在当前文档明写。skill 目前只能在文字里标注待确认，没有统一的事实/假设/继承来源结构，后续复核成本较高。
- 缺少自动沉淀“本次执行问题”的固定输出块：用户明确要求总结处理中遇到的问题，这类复盘对迭代 skill 很有价值，但当前 skill 模板没有内置“执行问题总结/skill 建议”章节。

## 对 by-slg-testcase 的迭代建议

- 把“执行中遇到的问题”和“对 skill 的建议”做成可选标准章节，方便每次任务反哺 skill。
