# Feature Breakdown

当功能较大、描述较模糊，或者测试边界不清楚时，先做功能拆分，再开始写测试。

## 目标

把一个功能拆成适合测试设计的更小行为单元，明确主流程、主状态链路、界面范围、测试点来源和阻塞问题。

## 拆分步骤

1. 读取需求文档，关注用户目标、业务逻辑、关键规则和页面范围，生成主流程。
2. 总结功能的主状态链路，明确状态流转和关键分支。
3. 按照主流程和主状态链路，分析一共有几个页面、弹窗、子页面或关键模块。
4. 对每个页面或模块继续做测试点拆分。
5. 先生成结构化的测试点分析，再基于测试点分析生成测试用例。

## 拆分方式

在完成主流程、主状态链路和界面拆分后，需要继续对每个页面或模块做测试点拆分。

拆分时优先按“用户可感知行为”和“需求规则”拆，不要按代码文件或技术实现拆。

每个页面或模块，至少检查以下维度是否需要覆盖：

1. 展示维度
- 页面元素是否展示完整。
- 文案、图片、图标、默认态是否正确。
- 布局、层级、显隐是否符合需求。

2. 输入维度
- 是否可为空。
- 长度限制、格式限制、字符限制是否明确。
- 特殊符号、空格、大小写、默认值、占位文案、可编辑性是否有规则。

3. 操作维度
- 点击、切换、勾选、展开、关闭、返回等操作是否生效。
- 按钮是否可点击，是否存在防重复点击或防重复提交。
- 操作路径、跳转路径、返回路径是否正确。

4. 状态维度
- 是否存在未开始、进行中、已完成、失败、过期、禁用、空态、加载态等状态。
- 不同状态下，页面元素、按钮、文案、奖励、跳转是否正确变化。

5. 交互联动维度
- 元素之间是否存在联动关系。
- 前置条件变化后，按钮状态、页面状态、提示信息是否同步变化。
- 页面与弹窗之间、父子页面之间的交互是否正确。

6. 规则校验维度
- 正向场景是否成立。
- 逆向场景是否被拦截。
- 边界值、异常值、冲突值是否被正确处理。
- 规则是否会因不同条件组合而产生不同结果。

7. 结果反馈维度
- 成功后的提示、跳转、刷新、奖励发放是否正确。
- 失败后的报错、toast、弹窗、错误文案是否正确。
- 操作后状态是否即时更新，返回后是否保持一致。

8. 异常与边界维度
- 空数据、超长、非法值、重复提交、弱网、超时、中断等情况是否正确处理。
- 用户前后空格、大小写、特殊输入、快速重复操作等边界是否覆盖。

9. 权限与前置条件维度
- 是否满足解锁条件、等级条件、资格条件、勾选条件或资源条件。
- 不满足前置条件时是否正确拦截，并给出正确提示。

10. 数据一致性维度
- 页面展示、记录、排行、弹窗、邮件等多处数据口径是否一致。
- 操作前后数据是否正确刷新、持久化、补发或结算。

不是每个功能都必须覆盖所有维度，要按需求实际内容选择适用维度；没有在需求中体现的维度，不要强行补成测试点。

## 测试点分析输出要求

1. 在生成最终测试用例前，必须先输出“测试点分析”。
2. 测试点分析至少包含：所属模块、测试点。
3. 测试点要从界面元素、规则、状态、交互和结果中逐项展开，不要直接用页面级汇总语句代替。
4. 如果一个测试点同时覆盖多个元素、多个规则、多个状态或多个结果，需要继续拆分。
5. 测试点描述要使用完整句式，至少包含“谁/什么场景下，做什么，结果应怎样”。

## 测试点规则

1. 测试点的依据是需求文档中的具体规则。
2. 文档没有明确规则时，不要擅自补规则；如果会影响测试点拆分方向，先整理为待确认项。
3. 待确认项要在生成最终测试用例前暴露，不要只放在最终文档末尾。
4. 不同页面之间的切换、相关入口跳转、返回路径，也需要作为测试点拆分。
5. 页面中的关键元素都需要判断是否要拆分，例如按钮、输入框、选项、展示位、说明文字、奖励区、状态标签等。
6. 拆分时优先保证主链路完整，再补充分支、逆向、异常和边界场景。
7. 页面级汇总描述不能直接作为测试用例。
8. 并列元素、并列规则、并列状态、并列结果要拆成多条测试点。

## 阻塞规则

1. 如果需求存在冲突、缺失，且会影响测试点拆分方向，需要先提出问题，再继续输出最终测试用例。
2. 阻塞问题优先按“影响主链路 > 影响状态判断 > 影响结果校验”的顺序整理。
3. 不要对未定义的业务规则做假设。
4. 阻塞问题要在最终测试用例前暴露，不要藏到文档末尾。

## 测试用例规则

1. 一条用例只包含一个测试点和对应的期望结果，用一句话总结。
2. 测试用例可以按所属模块排序整理，但最终输出必须汇总成一张统一表格，不要按模块拆成多张表。
3. 测试用例是针对测试点的验证展开，不是把测试点直接改写成一句用例。
4. 测试用例应该同时覆盖正向、逆向、边界、状态变化、结果反馈等适用验证场景，因此一个测试点通常需要拆成多条用例进行验证。
5. 测试用例数量必须远多于测试点数量；如果最终数量接近 1:1，说明验证展开不足，需要继续补充。
6. 最终测试用例表只需要包含：用例编号、所属模块、标题（测试用例本体）、优先级。
7. 不要在最终测试用例表中增加“对应测试点”“用例类型”等额外列。
8. 测试用例必须逐条对应测试点分析，不要跳过映射过程。
9. 测试用例标题要使用完整句式，至少包含“谁/什么场景下，做什么，结果应怎样”。

## 粒度自检

在输出最终用例前，检查以下问题：

- 是否还有一条用例同时覆盖多个页面元素。
- 是否还有一条用例同时覆盖多个规则。
- 是否还有一条用例同时覆盖多个状态。
- 是否还有一条用例同时覆盖多个结果。
- 测试用例总数是否已经远多于测试点总数；如果数量接近，说明还停留在测试点改写层，需要继续展开验证场景。
- 最终测试用例输出是否只有一张统一表格，且字段是否只保留“用例编号、所属模块、标题、优先级”。
- 标题里如果出现“和、及、并、并且、或、/、多个”等并列结构，优先判断是否需要继续拆分。

## 约束

- 优先按用户行为和需求规则拆分，不要只按代码文件拆分。
- 不要把拆分结果写成纯技术实现步骤。
- 不要把文档未定义的规则伪装成测试点，不确定可以随时提问。
