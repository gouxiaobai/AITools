# BY SLG Testcase

根据用户提供的需求文档、活动说明、表格、截图或文本描述，输出可执行的 SLG 游戏测试用例。优先基于证据提取规则，不要把假设写成已确认事实。默认输出应兼容 `BM测试用例库_基础用例_导入模板-先锋.xlsx` 的导入列结构。

## 工作原则

- 先提取已确认事实，再补充通用风险。
- 先确定主事实来源，再决定哪些附件只作为补充证据，不要无差别通读所有材料。
- 如果附件内容无法可靠识别，只输出可确认部分，并在“需要补充的信息”中列出缺口。
- 不要为了凑数量编造数值上限、状态规则、奖励内容或时间规则。
- 默认覆盖主流程、异常流程、边界流程；涉及多人交互、排行、战斗结算、奖励发放、联盟协作、活动结算时，额外覆盖并发、幂等、断线重连、重复请求和越权风险。
- 测试步骤必须可执行，避免抽象描述。

## 材料优先级与读取策略

- 优先读取 Markdown、需求文本、策划说明、已整理的规则表；这些通常是主事实来源。
- 再读取结构化表格，只提取和当前玩法直接相关的 sheet、字段、样例行，不要把整份配表机械搬进上下文。
- 截图、长图、录屏、美术稿优先用于补充界面状态、动画表现、入口位置和交互路径；不要仅凭视觉稿臆造服务端规则。
- 如果目录里同时存在“总活动文档 + 子活动目录 + 配表 + 图片/录屏”，默认先读总活动文档，再按缺口补读子活动和配表。
- 当材料量很大时，先输出“已确认/待确认/风险”，再决定是否继续深挖，不要为了完整性拖慢首轮产出。

## Windows 路径注意事项

- 在 Windows 环境下，读取中文路径文件时始终显式使用 UTF-8 读取文本。
- 如果内联执行方式在中文路径上出现乱码或路径变成问号，不要继续沿用同一方式；优先改用工作区中的脚本文件、`pathlib` 枚举、或 PowerShell 原生命令读取。
- 对 Excel、图片、视频等二进制文件，先验证文件是否可枚举、可定位，再进行内容提取。
- 如果某个附件因路径或编码问题无法稳定读取，要把这个问题写入“需要补充的信息”或“执行中遇到的问题”，不要静默忽略。

## 执行流程

1. 盘点输入材料，按“主事实来源 / 补充证据 / 暂不读取”分类。
2. 识别输入材料类型，提取已确认的入口、条件、限制、数值、状态、奖励、消耗和时间规则。
3. 列出“需求确认项”：
   - 已确认
   - 待确认
   - 继承逻辑
   - 基于 SLG 经验补充的风险
4. 列出“所属分组”拆解。如果模块边界不清，读取 `references/slg-modules.md`。
5. 如果是组合活动、多页签活动或“复用旧玩法”需求，先拆成：
   - 组合入口层
   - 子活动层
   - 通用复用层
   - 活动结束/补发层
6. 依据 `references/testcase-strategy.md` 设计测试点。
7. 依据 `references/import-template-mapping.md` 和 `references/testcase-template.md` 输出最终结果。

## 输出规则

- 默认输出列为：`标题* | 所属分组 | 维护人 | 前置条件 | 步骤描述 | 预期结果 | 用例等级 | 用例类型`
- 默认以“可导入表格”为主输出，不再强制输出旧版的 `用例ID / 模块 / 子功能 / 需求依据 / 假设/备注` 列。
- 如果用户没有明确提供维护人，`维护人` 留空，不要编造姓名。
- `所属分组` 使用层级路径，按“一级分组|二级分组|三级分组”填写；优先根据玩法入口、功能模块、子功能组织。
- `步骤描述` 与 `预期结果` 按模板要求一一对应；多步场景使用 `【1】`、`【2】`、`【3】` 编号并换行。
- `用例等级` 默认使用 `P0 / P1 / P2 / P3`。
- `用例类型` 只有在用户明确给出枚举或导入系统已有值时才填写；否则留空，避免导入失败。
- 只有在用户明确要求时，才额外输出“需求依据”“假设/备注”“是否自动化”“测试类型”等扩展字段。
- 测试类型如需输出，只使用：`Positive / Negative / Boundary / Stress / Concurrency / Robustness / Security`
- 如果用户要求保留分析信息，可在表格前补充“需求确认项”；否则优先只给导入表。
- 当规则明显来自“复用旧玩法”而不是当前文档明写时，把它放在“继承逻辑”中，不要冒充“已确认”。
- 如果某条用例依赖假设、行业经验或常见 SLG 风险补充，优先在表格外统一列出，不要破坏导入列结构。
- 如果用户要求复盘本次生成过程，或任务中确实暴露出明显障碍，可在表格后追加“执行中遇到的问题”“对 skill/流程的建议”。
- 如果用户只要求补充某个模块或某一批场景，只更新对应范围，不重写全量内容。
